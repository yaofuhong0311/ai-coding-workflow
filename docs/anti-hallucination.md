# 反幻觉规则设计

## 什么是 AI Coding 幻觉？

AI Coding Agent 的"幻觉"和 LLM 文本生成的幻觉类似，但表现形式不同：

- **编造文件路径**：`import from app/utils/helper.py` — 这个文件不存在
- **编造函数签名**：`client.send(message, timeout=30)` — 实际上没有 timeout 参数
- **编造 API 行为**：认为某个 API 返回 list，实际返回 dict
- **编造依赖版本**：引入项目中没有安装的包
- **忽略已有约定**：项目用 snake_case，Agent 写了 camelCase

这些不是"不会写代码"，而是 Agent 凭记忆推断而不是查看事实。

## 三条规则

### 规则 1: 读优先于写

**核心**：任何写操作之前，先读。

```
修改文件 → 先 read 确认当前内容
调用函数 → 先 grep 确认函数存在
写 import → 先 find 确认文件位置
引用模块 → 先读模块确认导出内容
```

为什么有效？因为大部分幻觉的根源是"凭记忆写代码"。强制 read-before-write 打断了这个模式。

### 规则 2: 不要编造

**核心**：不确定就查，不猜。

```
不确定函数签名 → 查看源码
不确定配置项 → 查看文档
不确定 API 行为 → 看实际响应
不确定时说 → "我需要先确认"
```

### 规则 3: 基于现有代码扩展

**核心**：先看别人怎么做的，再写。

```
新功能 → 先读同类模块的实现
新文件 → 看同级文件的命名和风格
新依赖 → 确认项目是否已在用
新类型 → 检查有没有现成的可复用
```

## 三层防御体系

之前的设计是两层：

```
反幻觉规则 = 前置过滤器（预防）
ruflo       = 后置安全网（兜底）
```

但这里有一个缺口：**规则是"建议"，不是"强制"。** Agent 可能忘记遵守规则，而 ruflo 只是事后记录，不能阻止错误发生。

加入 ast-grep 后，变成三层：

```
反幻觉规则 = 软约束（预防）     ← CLAUDE.md 里的行为指导，Agent 可能忘记
ast-grep   = 硬执行（拦截）     ← PostToolUse hook 机械扫描，不可绕过
ruflo      = 后置记录（兜底）   ← 记录行为和错误，用于蒸馏改进
```

### 软约束 vs 硬执行

| 维度 | 软约束（反幻觉规则） | 硬执行（ast-grep） |
|------|---------------------|-------------------|
| 载体 | CLAUDE.md 文本 | PostToolUse hook |
| 执行方式 | Agent 自觉遵守 | 机械扫描，违规注入上下文 |
| 可绕过 | 可以（Agent 忘了就跳过） | 不可以（同步 hook，写完自动触发） |
| 覆盖范围 | 广泛（行为模式） | 精确（8 类具体反模式） |
| 适合什么 | 通用行为准则 | 可机械判定的代码缺陷 |

ast-grep 覆盖的 8 类 Python 反模式：

| 规则 | 严重度 | 捕获什么 |
|------|--------|---------|
| `bare-except` | error | `except:` 不指定异常类型 |
| `mutable-default-arg` | error | `def foo(bar=[])` 可变默认参数 |
| `eval-exec-usage` | error | `eval()/exec()` 安全风险 |
| `empty-except-pass` | warning | `except: pass` 吞掉错误 |
| `blocking-in-async` | error | async 函数内用 `time.sleep()` |
| `star-import` | warning | `from x import *` 污染命名空间 |
| `hardcoded-credentials` | error | 硬编码密码/密钥/token |
| `print-in-production` | warning | 非测试文件中的 `print()` |

### 为什么需要硬执行？

软约束的问题：**Agent 最容易犯错的时候，恰恰是最容易忘记规则的时候。** 复杂任务中 Agent 专注于实现逻辑，"先读再写"这类规则会被优先级更高的目标挤掉。

硬执行不依赖 Agent 的自觉，而是在 Agent 产出代码后立即机械检查。违规信息直接注入 Agent 上下文，Agent 被迫看到并修复。

## 效果评估

这些规则是 semantic memory 矫正 procedural memory 的默认习惯：

- **概率性改善**：软约束不能 100% 消除幻觉，但能显著降低频率
- **确定性改善**：硬执行对 8 类反模式的拦截率为 100%
- **编造路径类错误**：上线后明显减少
- **编码风格一致性**：新增代码更贴合现有项目约定
- **需要持续优化**：根据实际犯错模式补充新的软规则和硬规则

## 设计哲学

> 与其让 AI 变得更"聪明"，不如让它变得更"谨慎"。

大部分 coding 错误不是能力不足，而是太自信——以为自己记得，实际记错了。

**软约束**解决的是意识问题：强制增加一个"确认"步骤。
**硬执行**解决的是遗忘问题：即使 Agent 忘了规则，机械检查仍然会拦截。

两者互补，不可互相替代。反幻觉规则仍然放在 `CLAUDE.md` 最前面，因为它覆盖的行为范围远大于 ast-grep 的 8 条规则。
