# 反幻觉规则设计

## 什么是 AI Coding 幻觉？

AI Coding Agent 的"幻觉"和 LLM 文本生成的幻觉类似，但表现形式不同：

- **编造文件路径**：`import from app/utils/helper.py` — 这个文件不存在
- **编造函数签名**：`client.send(message, timeout=30)` — 实际上没有 timeout 参数
- **编造 API 行为**：认为某个 API 返回 list，实际返回 dict
- **编造依赖版本**：引入项目中没有安装的包
- **忽略已有约定**：项目用 snake_case，Agent 写了 camelCase

这些不是"不会写代码"，而是 Agent 凭记忆推断而不是查看事实。

## 三条规则

### 规则 1: 读优先于写

**核心**：任何写操作之前，先读。

```
修改文件 → 先 read 确认当前内容
调用函数 → 先 grep 确认函数存在
写 import → 先 find 确认文件位置
引用模块 → 先读模块确认导出内容
```

为什么有效？因为大部分幻觉的根源是"凭记忆写代码"。强制 read-before-write 打断了这个模式。

### 规则 2: 不要编造

**核心**：不确定就查，不猜。

```
不确定函数签名 → 查看源码
不确定配置项 → 查看文档
不确定 API 行为 → 看实际响应
不确定时说 → "我需要先确认"
```

### 规则 3: 基于现有代码扩展

**核心**：先看别人怎么做的，再写。

```
新功能 → 先读同类模块的实现
新文件 → 看同级文件的命名和风格
新依赖 → 确认项目是否已在用
新类型 → 检查有没有现成的可复用
```

## 和 ruflo 的关系

```
反幻觉规则 = 前置过滤器（预防）
ruflo       = 后置安全网（兜底）
```

反幻觉规则在 Agent 生成代码之前就约束了行为模式——先读再写、不猜。

但规则不是 100% 有效的（Agent 可能"忘记"遵守）。ruflo 作为后置安全网，记录了 Agent 实际的行为和犯的错误，下次可以提醒。

## 效果评估

这些规则是 semantic memory 矫正 procedural memory 的默认习惯：

- **概率性改善**：不能 100% 消除幻觉，但能显著降低频率
- **编造路径类错误**：上线后明显减少
- **编码风格一致性**：新增代码更贴合现有项目约定
- **需要持续优化**：根据实际犯错模式补充新规则

## 设计哲学

> 与其让 AI 变得更"聪明"，不如让它变得更"谨慎"。

大部分 coding 错误不是能力不足，而是太自信——以为自己记得，实际记错了。强制增加一个"确认"步骤，就能避免大部分问题。

这也是为什么反幻觉规则放在 `CLAUDE.md` 的最前面——它是最基础、最重要的行为约束。
